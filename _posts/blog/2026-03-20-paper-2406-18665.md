## 論文概要（Abstract）

大規模言語モデル（LLM）のAPIコストと応答品質はトレードオフの関係にある。著者らは、クエリの難易度に応じて高品質・高コストの**強モデル**（例: GPT-4）と低コストの**弱モデル**（例: Mixtral-8x7B）を動的に振り分ける**RouteLLM**フレームワークを提案している。Chatbot Arenaの人間選好データからルーターを訓練し、品質をほぼ維持しながらコストを最大85%削減できたと報告されている。

## 背景と動機（Background & Motivation）

現在のLLMエコシステムでは、GPT-4やClaude Sonnetのような高性能モデルと、Mixtral-8x7BやLlama-3-8Bのような軽量モデルが共存している。高性能モデルは品質に優れるが、APIコストが10〜100倍高い。一方、多くのクエリ（単純な質問、定型応答等）は軽量モデルで十分な品質が得られる。

Zenn記事の構成では、LiteLLMの`routing_strategy: "least-busy"`で負荷分散を行っているが、これは**クエリの難易度を考慮しない**単純なロードバランシングである。RouteLLMは、クエリの内容を分析して最適なモデルを選択する**コンテンツアウェアなルーティング**を実現する。

著者らの問題定式化（論文Section 3より）：ルーターの目的は、品質制約$\epsilon$を満たしながらコストを最小化する関数$r: \mathcal{Q} \to \{0, 1\}$を学習することである。ここで$r(q) = 1$は強モデルへのルーティングを意味する。

## 技術的詳細（Technical Details）

### 問題定式化

$N$個のLLMモデル $\{M_1, M_2, ..., M_N\}$ があり、各モデルのクエリ$q$に対する品質を$Q_i(q)$、コストを$C_i(q)$とする。RouteLLMでは簡略化として、強モデル$M_s$と弱モデル$M_w$の2モデル間のルーティングを考える。

ルーター$r: \mathcal{Q} \to [0, 1]$は、クエリ$q$に対して強モデルを使う確率（スコア）を出力する。閾値$\tau$を設定し：

$$
\text{route}(q) = \begin{cases} M_s & \text{if } r(q) \geq \tau \\ M_w & \text{if } r(q) < \tau \end{cases}
$$

ここで$\tau$は**コスト-品質トレードオフを制御する唯一のハイパーパラメータ**である。$\tau$を上げるとコスト削減が増し、下げると品質が向上する。

### ルーターアーキテクチャ

著者らは3つのルーターアーキテクチャを比較検討している。

#### 1. Matrix Factorization Router

Chatbot Arenaのペア比較データを行列分解で学習する。各モデル$i$と各クエリの潜在特徴$q$を$d$次元ベクトルで表現し、勝率を内積で予測する：

$$
P(\text{model } i \text{ wins on } q) = \sigma(\mathbf{m}_i^T \mathbf{q}_j + b_i)
$$

ここで、
- $\mathbf{m}_i \in \mathbb{R}^d$: モデル$i$の潜在表現
- $\mathbf{q}_j \in \mathbb{R}^d$: クエリ$j$の潜在表現
- $b_i$: モデル$i$のバイアス項
- $\sigma$: シグモイド関数

訓練損失はBradley-Terryモデルに基づく対数尤度：

$$
\mathcal{L} = -\sum_{(i,j,q)} \left[ y_{ijq} \log P_{ijq} + (1 - y_{ijq}) \log(1 - P_{ijq}) \right]
$$

#### 2. BERT Classifier Router

クエリテキストをBERT系モデル（著者らの実装ではbert-base-uncased）でエンコードし、[CLS]トークンの表現を2クラス（強モデル/弱モデル）に分類する。

```python
import torch
import torch.nn as nn
from transformers import BertModel, BertTokenizer

class BERTRouterClassifier(nn.Module):
    """BERT-based LLM Router

    Args:
        model_name: 使用するBERTモデル名
        hidden_dim: 分類ヘッドの隠れ層次元
    """
    def __init__(self, model_name: str = "bert-base-uncased", hidden_dim: int = 128):
        super().__init__()
        self.bert = BertModel.from_pretrained(model_name)
        self.classifier = nn.Sequential(
            nn.Linear(768, hidden_dim),
            nn.ReLU(),
            nn.Dropout(0.1),
            nn.Linear(hidden_dim, 1),
            nn.Sigmoid()
        )

    def forward(self, input_ids: torch.Tensor, attention_mask: torch.Tensor) -> torch.Tensor:
        """クエリに対する強モデル使用確率を返す

        Args:
            input_ids: トークナイズされた入力 (batch_size, seq_len)
            attention_mask: アテンションマスク (batch_size, seq_len)

        Returns:
            強モデルを使用する確率 (batch_size, 1)
        """
        outputs = self.bert(input_ids=input_ids, attention_mask=attention_mask)
        cls_output = outputs.last_hidden_state[:, 0, :]  # [CLS] token
        return self.classifier(cls_output)
```

#### 3. Causal LM Router

Llama等のCausal LMをファインチューニングし、クエリに対して「この質問は強モデルが必要か？」を判定する。BERT分類器より高精度だが推論レイテンシが大きい。

### データ拡張

著者らは、Chatbot Arenaの比較データに加えて、以下のデータ拡張手法を適用している（論文Section 4.2より）：

1. **Golden-label augmentation**: GPT-4とMixtralの両方にクエリを送り、GPT-4（ジャッジ）で品質を比較。自動的にラベルを生成
2. **Threshold calibration**: バリデーションセットで$\tau$を調整し、目標コスト削減率を達成

## 実験結果（Results）

著者らはGPT-4 Turboを強モデル、Mixtral-8x7Bを弱モデルとして3つのベンチマークで評価を実施している。

| ベンチマーク | GPT-4単体 | RouteLLM (MF) | コスト削減率 | 品質差 |
|------------|----------|---------------|------------|--------|
| MT-Bench | 9.02 | 8.98 | 85%（論文Table 2より） | -0.04 |
| MMLU | 84.5% | 83.9% | 45%（論文Table 2より） | -0.6% |
| GSM8K | 94.2% | 93.8% | 35%（論文Table 2より） | -0.4% |

著者らの分析では、MT-Benchで高いコスト削減率が達成できる理由は、MT-Benchのクエリの多くが弱モデルでも十分な品質を出せる汎用的な質問であるためとされている。一方、GSM8Kは数学推論タスクであり、多くのクエリが強モデルを必要とするため削減率は低い。

**転移学習の結果**（論文Section 5.4より）：強モデルをGPT-4からClaude 3.5 Sonnetに変更しても、ルーターの性能はほぼ維持された（品質低下<1%）。これは、ルーターがクエリの「難易度」を学習しており、特定のモデルペアに過適合していないことを示唆している。

## 実運用への応用（Practical Applications）

Zenn記事のLiteLLM構成において、RouteLLMは以下のように活用できる：

1. **コスト制御の自動化**: LiteLLMのVirtual Keys（予算管理）とRouteLLMのコスト最適化ルーティングを組み合わせ、部署ごとの予算上限内で最適な品質を自動的に提供する
2. **ローカル/クラウドの使い分け**: 弱モデルにOllama（ローカル）、強モデルにクラウドAPI（OpenAI/Anthropic）を割り当て、機密データは自動的にローカルモデルで処理する
3. **段階的導入**: 最初は$\tau$を低く設定（品質優先）し、利用データの蓄積に応じて$\tau$を上げてコスト削減を進める

## 関連研究（Related Work）

- **FrugalGPT** (Chen et al., 2023): 複数LLMをカスケード的に呼び出し、品質スコア推定器で早期停止する手法。RouteLLMの1段ルーティングと比較して、カスケードは最悪ケースのレイテンシが増加する
- **LLM-Blender** (Jiang et al., 2023): 複数LLMの出力をアンサンブルする手法。品質は向上するが、全モデルを呼び出すためコスト削減にはならない
- **Hybrid LLM** (Ding et al., 2024): RouteLLMと類似のルーティングアプローチだが、訓練データにChatbot Arena選好データを使用しない点が異なる
