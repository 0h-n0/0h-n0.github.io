本記事は [Don't Break the Cache: An Evaluation of Prompt Caching for Long-Horizon Agentic Tasks](https://arxiv.org/abs/2601.06007) の解説記事です。

## 論文概要（Abstract）

本論文は、主要LLMプロバイダ（OpenAI、Anthropic、Google）が提供するプロンプトキャッシュ機能が、マルチターンのエージェントワークフローにおいてどの程度有効に機能するかを実証的に評価した研究である。著者らは500以上のエージェントセッションを対象にDeepResearch Benchを用いた実験を行い、適切なキャッシュ戦略の適用によりAPIコスト41〜80%削減、TTFT（Time to First Token）13〜31%改善が達成可能であることを報告している。

この記事は [Zenn記事: エージェントのプロンプトキャッシュ設計 — ツール定義と思考トークンを壊さない実装](https://zenn.dev/0h_n0/articles/37e71fbb85e1a6) の深掘りです。

## 背景と動機（Background & Motivation）

LLMエージェントの本番運用において、マルチターン会話とツール呼び出しの繰り返しによるAPIコストの急増は深刻な課題である。主要プロバイダはプロンプトキャッシュ（プレフィックスの再利用による計算省略）を提供しているが、エージェントワークフロー特有の動的な性質—ツール定義の変更、動的なtool_resultの挿入、セッション長の伸長—がキャッシュ設計を困難にしている。

著者らはこの問題について、「プロンプトキャッシュのエージェントワークロードにおける恩恵は未開拓のままである」と指摘し、体系的な評価フレームワークの必要性を提唱している。従来研究ではシングルターンや短いマルチターンでのキャッシュ効果は報告されていたが、長期間にわたるエージェントセッション（ウェブ検索、ツール呼び出し50回以上）での評価は存在しなかった。

## 主要な貢献（Key Contributions）

- **貢献1**: 3つのプロバイダ（OpenAI、Anthropic、Google）にまたがる包括的なキャッシュ戦略比較を初めて実施
- **貢献2**: 3つのキャッシュ配置戦略（フルコンテキスト、システムプロンプトのみ、動的ツール結果除外）の効果を定量化
- **貢献3**: プロンプトサイズ（500〜50,000トークン）とツール呼び出し回数（3〜50）に対するスケーリング特性を明らかにした

## 技術的詳細（Technical Details）

### プロンプトキャッシュの動作原理

プロンプトキャッシュはプレフィックスの完全一致に基づいて動作する。リクエスト$R_t$のプロンプトを$P_t = [p_1, p_2, \ldots, p_n]$（トークン列）とすると、キャッシュヒットの条件は以下の通りである：

$$
\text{cache\_hit}(R_t) = \begin{cases} 1 & \text{if } \exists k \geq k_{\min}: P_t[1:k] = P_{t-1}[1:k] \\ 0 & \text{otherwise} \end{cases}
$$

ここで、
- $P_t[1:k]$: リクエスト$t$のプロンプトの先頭$k$トークン
- $k_{\min}$: プロバイダごとの最小キャッシュ閾値（OpenAI: 1024トークン、Anthropic: 1024トークン、Google: プロンプト全体）

### 3つのキャッシュ戦略

著者らは以下の3つの戦略を評価している：

**戦略1: Full Context Caching（フルコンテキスト）**

全プロンプト内容（ツール定義 + システムプロンプト + 全会話履歴 + tool_result）にキャッシュブレークポイントを設定する方法。

```python
# Full Context: 全内容をキャッシュ対象とする
messages = [
    {"role": "system", "content": system_prompt, "cache_control": {"type": "ephemeral"}},
    *conversation_history,
    {"role": "user", "content": latest_query, "cache_control": {"type": "ephemeral"}},
]
```

**戦略2: System Prompt Only Caching（システムプロンプトのみ）**

システムプロンプトとツール定義のみにキャッシュを限定し、動的な会話履歴は毎回再計算させる方法。

**戦略3: Exclude Dynamic Tool Results（動的結果除外）**

会話履歴はキャッシュ対象に含めるが、tool_resultの動的部分（API応答、検索結果）はキャッシュ対象から除外する方法。

### プロバイダごとのキャッシュ挙動の差異

著者らの実験から、プロバイダ間で以下の重要な差異が明らかになった：

| プロバイダ | 最小トークン閾値 | キャッシュ粒度 | TTL | 課金体系 |
|-----------|:---:|:---:|:---:|:---:|
| Anthropic | 1024 | 128トークン単位 | 5分（デフォルト）/1時間 | write: 1.25x, read: 0.1x |
| OpenAI | 1024 | 128トークン単位 | 自動管理 | write: 無料, read: 0.5x |
| Google | プロンプト全体 | コンテキスト全体 | 1時間 | write: 無料, read: 0.25x |

### アブレーション実験の設計

著者らは以下の2軸でアブレーション実験を実施している：

1. **プロンプトサイズ変動**: 500, 1000, 5000, 10000, 50000トークンのシステムプロンプトでキャッシュ効果を測定
2. **ツール呼び出し回数変動**: 3, 10, 20, 50回のツール呼び出しを含むセッションでキャッシュヒット率を測定

コスト削減率$S$は以下の式で計算される：

$$
S = 1 - \frac{C_{\text{cached}}}{C_{\text{no\_cache}}} = 1 - \frac{T_{\text{read}} \cdot r_{\text{read}} + T_{\text{write}} \cdot r_{\text{write}} + T_{\text{new}} \cdot r_{\text{input}}}{(T_{\text{read}} + T_{\text{write}} + T_{\text{new}}) \cdot r_{\text{input}}}
$$

ここで、
- $T_{\text{read}}$: キャッシュから読み込まれたトークン数
- $T_{\text{write}}$: キャッシュに書き込まれたトークン数
- $T_{\text{new}}$: 新規計算されたトークン数
- $r_{\text{read}}, r_{\text{write}}, r_{\text{input}}$: 各カテゴリの単価

## 実験結果（Results）

### コスト削減効果

著者らの実験結果（論文Table 1相当）によると、各戦略・プロバイダの組み合わせでの削減率は以下の通りである：

| 戦略 | Anthropic | OpenAI | Google |
|------|:-:|:-:|:-:|
| Full Context | 80% | 62% | 41% |
| System Only | 45% | 38% | 30% |
| Exclude Dynamic | 72% | 55% | 38% |

Full Context戦略が最も高い削減率を示すが、著者らは重要な注意点を指摘している：「Full Contextは逆説的にレイテンシを増加させる場合がある」。これは、キャッシュ書き込みコスト（writeプレミアム）がヒットしない場合に通常の入力コストを上回るためである。

### TTFT改善

| 戦略 | Anthropic | OpenAI | Google |
|------|:-:|:-:|:-:|
| Full Context | 31% | 22% | 13% |
| System Only | 18% | 15% | 10% |
| Exclude Dynamic | 27% | 19% | 12% |

### スケーリング特性

著者らは「キャッシュの恩恵はプロバイダ固有の最小閾値に線形にスケールする」と報告している。具体的には：

- システムプロンプトが1024トークン未満の場合、Anthropic/OpenAIではキャッシュが一切機能しない
- 10,000トークン以上のシステムプロンプトでは、コスト削減効果が最大化する
- ツール呼び出し回数が20回を超えると、Full Context戦略のヒット率が低下し始める（20ブロック窓の影響）

## 実装のポイント（Implementation）

### 推奨される実装パターン

著者らの知見に基づく推奨実装は以下の通りである：

```python
from anthropic import Anthropic

client = Anthropic()

def build_cached_agent_request(
    tools: list[dict],
    system_prompt: str,
    messages: list[dict],
) -> dict:
    """キャッシュ最適化されたエージェントリクエスト構築
    
    著者らの「Exclude Dynamic」戦略に基づく実装。
    動的なtool_resultはキャッシュブレークポイント外に配置する。
    """
    # ツール定義: 最後の要素にcache_controlを付与
    cached_tools = [*tools[:-1], {**tools[-1], "cache_control": {"type": "ephemeral"}}]
    
    # メッセージ: 静的部分（ユーザークエリ）にのみブレークポイント
    cached_messages = []
    for msg in messages:
        if msg["role"] == "user" and not _is_tool_result(msg):
            # ユーザーメッセージ（静的）: キャッシュ対象
            cached_messages.append({
                **msg,
                "content": [
                    {**msg["content"][0], "cache_control": {"type": "ephemeral"}}
                ] if isinstance(msg["content"], list) else [
                    {"type": "text", "text": msg["content"], "cache_control": {"type": "ephemeral"}}
                ],
            })
        else:
            # tool_result（動的）: キャッシュ対象外
            cached_messages.append(msg)
    
    return {
        "model": "claude-sonnet-4-6",
        "max_tokens": 4096,
        "tools": cached_tools,
        "system": [{"type": "text", "text": system_prompt, "cache_control": {"type": "ephemeral"}}],
        "messages": cached_messages,
    }


def _is_tool_result(msg: dict) -> bool:
    """メッセージがtool_resultを含むか判定"""
    if not isinstance(msg.get("content"), list):
        return False
    return any(block.get("type") == "tool_result" for block in msg["content"])
```

### 設計上の注意点

1. **動的コンテンツは末尾に配置**: 変化する要素（tool_result、最新のユーザー入力）をプロンプト末尾に集約し、プレフィックスの安定性を確保する
2. **ツール定義を固定**: MCPで動的にツールが追加される場合でも、APIに渡すツール配列は固定し、`allowed_tools`で制御する
3. **ターン間隔の管理**: Anthropicの5分TTLを前提に、ユーザー応答待ち時間が5分を超える場合はpre-warmリクエストを検討する

## 実運用への応用（Practical Applications）

本論文の知見は、以下のプロダクションシナリオに直接適用可能である：

- **カスタマーサポートBot**: 長いセッション（20ターン以上）でFull Context戦略を採用し、20ブロック窓に注意した中間ブレークポイント配置
- **コード生成エージェント**: ツール定義が固定的なため、System Only戦略でも十分な削減効果が得られる
- **リサーチエージェント**: Web検索結果（動的）を含むため、Exclude Dynamic戦略が最適

著者らは「安定したプレフィックスを持つワークロードに限定される」と明確に制約を述べており、MCPベースの動的ツールセットシステムではキャッシュ恩恵が減少する点に注意が必要である。

## Production Deployment Guide

### AWS実装パターン（コスト最適化重視）

本論文の知見を活用し、プロンプトキャッシュ最適化済みのLLMエージェントをAWS上にデプロイする構成を示す。

| 規模 | 月間リクエスト | 推奨構成 | 月額コスト | 主要サービス |
|------|:---:|:---:|:---:|:---:|
| **Small** | ~3,000 (100/日) | Serverless | $50-150 | Lambda + Bedrock + DynamoDB |
| **Medium** | ~30,000 (1,000/日) | Hybrid | $300-800 | Lambda + ECS Fargate + ElastiCache |
| **Large** | 300,000+ (10,000/日) | Container | $2,000-5,000 | EKS + Karpenter + EC2 Spot |

**Small構成の詳細** (月額$50-150):
- **Lambda**: 1GB RAM, 60秒タイムアウト ($20/月)
- **Bedrock**: Claude 3.5 Haiku, Prompt Caching有効 ($80/月)
- **DynamoDB**: On-Demand, キャッシュメタデータ保存 ($10/月)
- **CloudWatch**: 基本監視 ($5/月)

**コスト削減テクニック**:
- Prompt Caching有効化で30-90%削減（本論文の知見を直接活用）
- Bedrock Batch API使用で50%削減（非リアルタイム処理）
- Spot Instances使用で最大90%削減（EKS + Karpenter）

**コスト試算の注意事項**: 上記は2026年5月時点のAWS ap-northeast-1（東京）リージョン料金に基づく概算値です。実際のコストはトラフィックパターン、バースト使用量により変動します。最新料金は[AWS料金計算ツール](https://calculator.aws/)で確認してください。

### コスト最適化チェックリスト

- [ ] Prompt Caching有効化（Anthropic: cache_control設定、OpenAI: 自動）

- [ ] 動的コンテンツをプロンプト末尾に配置

## 関連研究（Related Work）

- **CachedAttention (2407.01219)**: マルチターン会話でKVキャッシュをCPU/GPU階層に永続化する手法。本論文がAPI層での評価であるのに対し、CachedAttentionはサーバ内部の実装に焦点を当てている。
- **ChunkAttention (2311.04934)**: 共通プレフィックスのKVキャッシュをtrie構造で管理し、メモリ使用量30%削減を達成。本論文のプレフィックスマッチング概念の低レイヤー実装に相当する。
- **SGLang RadixAttention**: プレフィックスをRadix tree（基数木）で管理し、複数リクエスト間でKVキャッシュを効率的に共有する推論フレームワーク。

## まとめと今後の展望

本論文は、プロンプトキャッシュがエージェントワークフローにおいて41〜80%のコスト削減と13〜31%のレイテンシ改善をもたらすことを実証した。ただし著者らは、「ナイーブなFull Context戦略は逆説的にレイテンシを増加させうる」と警告しており、ワークロード特性に応じた戦略選択が不可欠である。今後は、MCPベースの動的ツールセットへの対応や、マルチモデル切り替え時のキャッシュ共有が研究課題として残されている。
