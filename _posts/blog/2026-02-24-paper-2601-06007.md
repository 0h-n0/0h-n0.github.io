## 論文概要（Abstract）

本論文は、OpenAI・Anthropic・Googleの3社が提供するプロンプトキャッシュ機能を、マルチターンエージェントタスクの統一ベンチマーク「DeepResearchBench」で比較評価した研究である。著者らは、500以上のエージェントセッションで3つのキャッシュ戦略（フルコンテキスト・システムプロンプトのみ・ツール結果除外）を比較し、APIコスト41–80%削減、TTFT（Time-To-First-Token）13–31%改善を報告している。

## 背景と動機（Background & Motivation）

LLMのAPI利用コストはトークン数に比例して増加し、マルチターンのエージェントタスクではシステムプロンプト・ツール定義・会話履歴が繰り返しリクエストに含まれるため、コストの累積が問題となる。主要プロバイダー3社はいずれもプロンプトキャッシュ（KVキャッシュの再利用）機能を提供しているが、API設計が異なるため、どのキャッシュ戦略がどのワークロードで有効かについて体系的な評価が存在しなかった。

著者らはこの課題に対して、マルチターンエージェントタスク（自律的にWeb検索ツールを使って調査を行うタスク）を用いた統一評価フレームワークを構築し、プロバイダー間のキャッシュ効果の違いを定量的に明らかにすることを目指している。

## 主要な貢献（Key Contributions）

- **貢献1**: マルチターンエージェントタスクにおけるプロンプトキャッシュの統一評価フレームワーク「DeepResearchBench」の構築。500以上のエージェントセッション、10,000トークンのシステムプロンプト、500–50,000トークンのプロンプトサイズ、3–50回のツール呼び出しを網羅する
- **貢献2**: 3つのキャッシュ戦略（フルコンテキスト・システムプロンプトのみ・ツール結果除外）の定量比較。プロバイダーごとに最適な戦略が異なることを実測データで示した
- **貢献3**: フルコンテキストキャッシュが逆効果になるケース（GPT-4oでTTFT 8.8%悪化）の発見。「キャッシュは多いほどよい」という直感的仮定に対する反例を提示した

## 技術的詳細（Technical Details）

### プロンプトキャッシュの基本原理

プロンプトキャッシュは、LLMの推論時にTransformerのAttention計算で生成されるKV（Key-Value）テンソルをサーバー側に保存し、同一プレフィックスの後続リクエストで再利用する仕組みである。プレフィックスが一致する部分のKV計算をスキップできるため、TTFTの短縮とコスト削減が実現される。

KVキャッシュの再利用条件は**プレフィックス完全一致**であり、先頭から連続してトークンが一致する部分のみが再利用対象となる。数式で表現すると、リクエスト$r$のプロンプトトークン列を$\mathbf{t}_r = (t_1, t_2, \ldots, t_n)$とし、キャッシュされたトークン列を$\mathbf{t}_c = (t_1', t_2', \ldots, t_m')$とすると、キャッシュヒット長$L$は以下で定義される。

$$
L = \max\{k \mid t_i = t_i' \text{ for all } i \leq k\}
$$

ここで、
- $L$: キャッシュヒット長（再利用可能なトークン数）
- $\mathbf{t}_r$: 新規リクエストのトークン列
- $\mathbf{t}_c$: キャッシュ済みトークン列

キャッシュヒット率は$\frac{L}{n}$で計算され、この値が高いほどコスト削減とTTFT改善が大きくなる。

### 評価フレームワーク: DeepResearchBench

著者らが構築したDeepResearchBenchは、マルチターンエージェントタスクに特化したベンチマークである。エージェントは自律的にWeb検索ツールを呼び出して調査を行い、複数ターンにわたってプロンプトが拡張されていく。このタスク構造により、以下の特性を持つプロンプトキャッシュの評価が可能となる。

1. **静的コンテンツ**: システムプロンプト（10,000トークン）、ツール定義
2. **動的コンテンツ**: ツール呼び出し結果（Web検索結果）、会話履歴
3. **プロンプトサイズの変動**: ターンが進むにつれ500→50,000トークンへ増加
4. **ツール呼び出し頻度**: セッションあたり3–50回

### 3つのキャッシュ戦略

論文では以下の3戦略を比較している。

**戦略A: フルコンテキストキャッシュ**
プロンプト全体（システムプロンプト＋ツール定義＋会話履歴＋ツール結果）をキャッシュ対象とする。キャッシュ書込量が最大となるが、ツール結果が動的に変化するためキャッシュヒット率が低下する可能性がある。

**戦略B: システムプロンプトのみキャッシュ**
変更頻度が最も低いシステムプロンプト部分のみをキャッシュ対象とする。キャッシュヒット率は安定するが、コスト削減の上限がシステムプロンプトの比率に制約される。

**戦略C: ツール結果除外キャッシュ**
プロンプト全体からツール呼び出し結果（動的コンテンツ）を除外してキャッシュする。戦略Aと戦略Bの中間的なアプローチである。

```python
from dataclasses import dataclass
from typing import Literal


@dataclass
class CacheStrategy:
    """プロンプトキャッシュ戦略の定義"""
    name: Literal["full_context", "system_only", "exclude_tool_results"]
    cached_components: list[str]

    def compute_cache_ratio(
        self,
        system_tokens: int,
        tool_def_tokens: int,
        history_tokens: int,
        tool_result_tokens: int,
    ) -> float:
        """キャッシュ対象トークンの割合を計算する。

        Returns:
            キャッシュ対象トークン / 全トークンの比率
        """
        total = system_tokens + tool_def_tokens + history_tokens + tool_result_tokens
        cached = 0
        if "system" in self.cached_components:
            cached += system_tokens
        if "tool_def" in self.cached_components:
            cached += tool_def_tokens
        if "history" in self.cached_components:
            cached += history_tokens
        if "tool_results" in self.cached_components:
            cached += tool_result_tokens
        return cached / max(total, 1)


# 3つの戦略を定義
STRATEGIES = {
    "A": CacheStrategy("full_context", ["system", "tool_def", "history", "tool_results"]),
    "B": CacheStrategy("system_only", ["system"]),
    "C": CacheStrategy("exclude_tool_results", ["system", "tool_def", "history"]),
}
```

## 実装のポイント（Implementation）

### プロバイダー別のキャッシュ制御

3社のAPIは以下のようにキャッシュ制御方式が異なり、戦略の実装方法がプロバイダーごとに変わる。

**OpenAI**: 1024トークン以上のプロンプトに対し自動でキャッシュが適用される。開発者は明示的にキャッシュを制御できない（2025年後半に追加された`prompt_cache_retention="24h"`パラメータを除く）。戦略Bを実現するには、プロンプト構造を工夫し、静的コンテンツを先頭に配置する必要がある。

**Anthropic**: `cache_control`パラメータで最大4つのブレークポイントを明示的に配置できる。TTLを5分と1時間から選択可能。戦略A/B/Cいずれも直接的に実装できる。

**Gemini**: 暗黙キャッシュ（自動）と明示的キャッシュ（キャッシュオブジェクトAPI）の2方式。明示的キャッシュではTTLを秒単位で自由設定できるが、ストレージ課金が発生する。

### 実装上の注意点

論文の知見から導かれる実装上のポイントは以下の通りである。

1. **プロンプト構造の固定化**: システムプロンプトにタイムスタンプやセッションIDを含めるとキャッシュが毎回無効化される。動的要素はプロンプト末尾に配置する
2. **ツール呼び出し結果の分離**: エージェントシステムでは、ツール呼び出し結果をキャッシュ対象から分離する設計が有効。Anthropicではブレークポイント配置、OpenAIではプロンプト構造の工夫が必要
3. **キャッシュヒット率のモニタリング**: 各プロバイダーのレスポンスに含まれるキャッシュメトリクス（cached_tokens、cache_read_input_tokens等）を監視し、期待どおりのヒット率が得られているかを継続的に確認する

**Small構成の詳細**（月額$50-150、2026年2月時点のap-northeast-1概算）:

## 実験結果（Results）

著者らは500以上のエージェントセッションを用いて3社のプロンプトキャッシュを評価している。以下に主要な結果を示す（論文Table 1およびFigure 3より）。

| モデル | 最適キャッシュ戦略 | コスト削減率 | TTFT改善率 |
|--------|-------------------|------------|-----------|
| GPT-5.2 | ツール結果除外（C） | 79.6% | 13.0% |
| Claude Sonnet 4.5 | システムプロンプトのみ（B） | 78.5% | 22.9%* |
| Gemini 2.5 Pro | システムプロンプトのみ（B） | 41.4% | 6.1% |
| GPT-4o | システムプロンプトのみ（B） | 45.9% | 30.9%* |

（*は統計的に有意なTTFT改善（p < 0.05）、出典: 論文Table 1）

### 戦略別の安定性

著者らの分析によると、戦略B（システムプロンプトのみ）が3社すべてで最も安定した効果を発揮している。一方、戦略A（フルコンテキスト）はGPT-4oでTTFTが8.8%悪化するケースが報告されている。これは、動的なツール呼び出し結果のキャッシュ書込オーバーヘッドが、読取の恩恵を上回ったことが原因と著者らは分析している。

### コスト削減率のプロバイダー間差異

Gemini 2.5 Proのコスト削減率（41.4%）がOpenAI（79.6%）・Anthropic（78.5%）と比較して低い理由について、著者らはGeminiのストレージ課金モデルとキャッシュ書込コストの構造的差異を指摘している。Geminiの明示的キャッシュでは時間ベースのストレージ課金が発生するため、短時間で多数のリクエストを処理するエージェントタスクでは相対的にコスト削減効果が薄くなる。

## 実運用への応用（Practical Applications）

本論文の知見は、マルチターンエージェントシステムの設計に直接適用できる。

**推奨される実装パターン**:
1. **エージェントシステム**: ツール呼び出しがある場合、OpenAIでは戦略C（ツール結果除外）、Anthropic・Geminiでは戦略B（システムプロンプトのみ）が最適
2. **RAGシステム**: 検索結果（動的コンテンツ）をプロンプト末尾に配置し、システムプロンプトのキャッシュヒット率を維持する
3. **マルチプロバイダー運用**: プロバイダーごとに最適戦略が異なるため、キャッシュ制御ロジックをゲートウェイ層で統一管理する設計が有効

**スケーリングの留意点**: セッション数が増加するとキャッシュの競合が発生する可能性がある。OpenAIの`prompt_cache_key`パラメータやAnthropicのブレークポイント設計を活用し、同一プレフィックスのリクエストを効率的にルーティングする仕組みが必要となる。

## 関連研究（Related Work）

- **Prompt Cache（arxiv:2311.04934）**: プロンプトを「モジュール」として定義し、Attention Stateの事前計算・再利用を実現するシステム。本論文が評価するAPI レベルのキャッシュ機能の理論的基盤
- **SGLang/RadixAttention（arxiv:2312.07104）**: Radix Treeを使ったKVキャッシュの自動共有機構。プレフィックスマッチングの実装として本論文の議論と直接関連
- **PagedAttention/vLLM（SOSP 2023）**: OSの仮想メモリ技術を応用したKVキャッシュのページ管理。プロンプトキャッシュの基盤となるメモリ管理手法

## まとめと今後の展望

本論文は、プロンプトキャッシュの「どの戦略をどのプロバイダーで使うべきか」という実践的な問いに対して、統一ベンチマークによる定量的な回答を提示した。著者らが報告した主要な知見として、（1）システムプロンプトのみキャッシュが最も安定した効果を発揮すること、（2）フルコンテキストキャッシュは逆効果になるリスクがあること、（3）プロバイダーごとに最適戦略が異なることが挙げられる。

今後の研究方向として、より長いセッション（100ターン以上）での評価、マルチモーダル入力（画像・音声）を含むエージェントタスクでのキャッシュ効果、およびプロバイダー側のキャッシュ実装の最適化が期待される。
