本記事は [Prompt Cache: Modular Attention Reuse for Low-Latency Inference](https://arxiv.org/abs/2311.04934) の解説記事です。

## 論文概要（Abstract）

著者らは、LLMの推論時にプロンプトの共有部分（システムプロンプト、Few-shotサンプル等）のアテンション状態（KVキャッシュ）を事前計算・保存し、後続のリクエストで再利用するPrompt Cacheシステムを提案している。プロンプトを「Prompt Module」と呼ばれるモジュール単位に分割することで、プレフィックスの完全一致だけでなく部分一致でもキャッシュヒットを可能にする設計が特徴である。著者らは、LLaMA-2-13B/70Bでの評価でTTFT（Time-To-First-Token）を最大8倍削減したと報告している。

この記事は [Zenn記事: LangGraph×Claude Sonnet 4.6のプロンプトキャッシュ最適化でAgentic RAGコスト90%削減](https://zenn.dev/0h_n0/articles/555a4e799660de) の深掘りです。

## 背景と動機（Background & Motivation）

Transformer系LLMの推論では、入力プロンプトの各トークンに対してAttention演算（QKV計算）を実行し、KV（Key-Value）テンソルを生成する。このプレフィル（prefill）フェーズはプロンプト長に比例して計算コストが増大し、特にシステムプロンプトやFew-shotサンプルが長い場合にTTFTのボトルネックとなる。

従来のKVキャッシュ最適化（PagedAttention等）はメモリ管理の効率化に焦点を当てていたが、**同一または類似のプロンプトプレフィックスが異なるリクエスト間で繰り返し計算される問題**には対処していなかった。著者らは、多くのLLMアプリケーション（チャットBot、コードアシスタント、RAGシステム等）で共通するシステムプロンプトやツール定義が繰り返し使用される点に着目し、これらのKVキャッシュを永続化・再利用する手法を提案した。

## 主要な貢献（Key Contributions）

- **Prompt Moduleの概念導入**: プロンプトをモジュール単位に分割し、各モジュールのKVキャッシュを独立に管理・再利用可能にした
- **位置エンコーディングの独立性**: 各Prompt Moduleが位置独立なKVキャッシュを持つことで、モジュールの組み合わせが変わってもキャッシュが再利用可能
- **既存モデルへの変更不要**: 推論時のみの最適化であり、モデルの再学習やアーキテクチャ変更を必要としない（pure inference-time optimization）
- **TTFT最大8倍削減**: LLaMA-2-13B/70Bでの実験で、プレフィルフェーズのレイテンシを大幅に改善

## 技術的詳細（Technical Details）

### Prompt Moduleの設計

著者らが提案するPrompt Moduleは、プロンプトの論理的な構成単位である。一般的なLLMプロンプトは以下のような階層構造を持つ。

```
[System Instruction] + [Few-shot Examples] + [Context/RAG Documents] + [User Query]
```

従来のプレフィックスキャッシュでは、先頭からの完全一致部分のみがキャッシュヒットの対象だった。著者らの手法では、各構成要素をPrompt Moduleとして独立に管理し、モジュール単位でのキャッシュ再利用を可能にしている。

```python
# Prompt Module定義の例（著者らの論文内表記をPythonで再現）
prompt_modules = {
    "system": PromptModule(
        id="sys-v1",
        content="You are a helpful coding assistant...",
        # このモジュールのKVキャッシュは永続化される
    ),
    "few_shot": PromptModule(
        id="fs-python-v1",
        content="Example 1: ...\nExample 2: ...",
    ),
    "context": PromptModule(
        id="ctx-dynamic",
        content=rag_retrieved_docs,  # 動的コンテンツ
    ),
}
```

### KVキャッシュの再利用メカニズム

標準的なTransformerのSelf-Attention計算は以下の通りである。

$$
\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V
$$

ここで、
- $Q \in \mathbb{R}^{n \times d_k}$: クエリ行列（$n$: シーケンス長、$d_k$: 次元数）
- $K \in \mathbb{R}^{n \times d_k}$: キー行列
- $V \in \mathbb{R}^{n \times d_v}$: バリュー行列

プレフィルフェーズでは、入力プロンプト全体のKVテンソルを計算する。Prompt Cacheでは、事前計算済みのPrompt Module $m$ に対応するKVテンソル $(K_m, V_m)$ をキャッシュから読み込み、新規トークンのみKVテンソルを計算する。

$$
K_{\text{full}} = [K_{m_1} \| K_{m_2} \| \cdots \| K_{m_k} \| K_{\text{new}}]
$$

$$
V_{\text{full}} = [V_{m_1} \| V_{m_2} \| \cdots \| V_{m_k} \| V_{\text{new}}]
$$

ここで、
- $K_{m_i}$: $i$番目のPrompt ModuleのキャッシュされたKテンソル
- $K_{\text{new}}$: 新規（キャッシュされていない）トークンのKテンソル
- $\|$: テンソルの連結（concat）

### 位置エンコーディングの取り扱い

多くのTransformerモデルはRotary Position Embedding（RoPE）を使用しており、位置エンコーディングがKVテンソルに埋め込まれている。著者らは2つのアプローチを提案している。

**アプローチ1: 位置非依存KVキャッシュ**

RoPEの回転行列 $R_{\theta, p}$ を分離し、位置非依存なKVテンソルを保存する。

$$
K_{\text{cache}} = W_K x \quad (\text{RoPE適用前})
$$

利用時に実際の位置に応じたRoPEを再適用する。

$$
K_{\text{positioned}} = R_{\theta, p} \cdot K_{\text{cache}}
$$

**アプローチ2: 固定位置KVキャッシュ**

各Prompt Moduleの開始位置を固定し、位置エンコーディング適用済みのKVキャッシュを保存する。モジュールの組み合わせが変わっても、各モジュールの位置が一定であればキャッシュを再利用できる。

### アルゴリズム

著者らが提案するPrompt Cacheの推論時のアルゴリズムを以下に示す。

```python
from dataclasses import dataclass
from typing import Optional
import torch


@dataclass
class PromptModule:
    """Prompt Moduleの定義"""
    module_id: str
    tokens: list[int]
    position_start: int  # 固定位置


class PromptCacheEngine:
    """Prompt Cache推論エンジン（著者らのアルゴリズムを再現）"""

    def __init__(self, model, cache_dir: str):
        self.model = model
        self.kv_store: dict[str, tuple[torch.Tensor, torch.Tensor]] = {}

    def prefill_with_cache(
        self,
        modules: list[PromptModule],
        new_tokens: list[int],
    ) -> tuple[torch.Tensor, torch.Tensor]:
        """キャッシュを活用したプレフィル処理

        Args:
            modules: 使用するPrompt Moduleのリスト
            new_tokens: キャッシュされていない新規トークン

        Returns:
            完全なKVテンソル (K_full, V_full)
        """
        cached_kv_list = []
        tokens_to_compute = []

        for module in modules:
            if module.module_id in self.kv_store:
                # キャッシュヒット: KVテンソルを読み込み
                cached_kv_list.append(self.kv_store[module.module_id])
            else:
                # キャッシュミス: 計算対象に追加
                tokens_to_compute.extend(module.tokens)

        # 新規トークンも計算対象
        tokens_to_compute.extend(new_tokens)

        # 新規トークンのKVテンソルを計算
        new_kv = self.model.compute_kv(tokens_to_compute)

        # キャッシュ済みKVと新規KVを連結
        all_k = torch.cat([kv[0] for kv in cached_kv_list] + [new_kv[0]], dim=1)
        all_v = torch.cat([kv[1] for kv in cached_kv_list] + [new_kv[1]], dim=1)

        return all_k, all_v
```

## 実験結果（Results）

著者らはLLaMA-2-13BおよびLLaMA-2-70Bを用いて評価を行っている（論文Section 5より）。

| 構成 | モデル | プロンプト長 | ベースラインTTFT | Prompt Cache TTFT | 高速化倍率 |
|------|-------|-----------|----------------|------------------|----------|
| A100 80GB | LLaMA-2-13B | 2048トークン | 320ms | 45ms | 7.1x |
| A100 80GB | LLaMA-2-13B | 4096トークン | 580ms | 72ms | 8.1x |
| A100 80GB | LLaMA-2-70B | 2048トークン | 890ms | 180ms | 4.9x |
| A100 80GB | LLaMA-2-70B | 4096トークン | 1650ms | 260ms | 6.3x |

著者らの報告によると、プロンプト長が長いほどキャッシュの効果が顕著になる。これは、キャッシュにより省略されるKV計算量がプロンプト長に比例するためである。

### キャッシュストレージのオーバーヘッド

著者らはキャッシュのメモリ・ストレージコストについても分析している。

- LLaMA-2-13B: 1024トークンのPrompt Moduleあたり約400MB（FP16精度）
- LLaMA-2-70B: 1024トークンのPrompt Moduleあたり約2GB（FP16精度）

GPU VRAM上にキャッシュを保持する場合はアクセスが最速だが、メモリ容量の制約がある。SSD/RAMへの永続化も可能だが、読み込みレイテンシが追加される。著者らは、頻繁に使用されるPrompt ModuleをVRAM上に、それ以外をSSD上に配置するティアード戦略を推奨している。

### 精度への影響

著者らは、Prompt Cacheの使用が生成品質に影響を与えないことを検証している。KVキャッシュの再利用は数値的に完全に等価な計算であり、浮動小数点の丸め誤差を除いて出力が同一であることが確認されている。

## 実装のポイント（Implementation）

### 最小キャッシュ可能トークン数

著者らは、キャッシュのオーバーヘッド（読み込み時間 + メモリ管理）が計算節約を上回らないためには、最低512トークン以上のPrompt Moduleサイズが推奨されると述べている。Anthropicのプロダクション実装では最小1024トークンが要件とされており、これは本論文の知見と整合する。

### Flash Attentionとの互換性

著者らはFlash Attention 2との組み合わせが可能であることを確認している。Flash AttentionのブロックレベルKV処理と、Prompt Cacheのモジュールレベルキャッシュは階層が異なるため、両者の恩恵を同時に受けることができる。

### Anthropic/OpenAIのプロダクション実装との関係

本論文のPrompt Module概念は、Anthropicの`cache_control`ブレークポイントやOpenAIの自動プレフィックスキャッシュの理論的基盤を形成している。Zenn記事で解説されている4層キャッシュブレークポイント設計（ツール定義→システムプロンプト→RAGコンテキスト→会話履歴）は、本論文のPrompt Moduleの実践的な適用パターンと見なすことができる。

## 実運用への応用（Practical Applications）

### Agentic RAGでの活用

Zenn記事で解説されているLangGraphのAgentic RAGパイプラインでは、各ステップ（ルーティング判定→検索→関連性評価→回答生成）で共通のツール定義とシステムプロンプトが使用される。本論文の知見に基づけば、これらの共通部分をPrompt Moduleとしてキャッシュすることで、マルチステップの推論コストを大幅に削減できる。

### コスト試算

プロンプト長が4096トークンのAgentic RAGパイプラインで5ステップの推論が発生する場合:

- **キャッシュなし**: 4096 × 5 = 20,480トークンを毎回処理
- **キャッシュあり**（CHR 80%想定）: 初回4096 + 以降819 × 4 = 7,372トークンを処理
- **削減率**: 約64%のトークン処理削減

Anthropic価格体系（キャッシュヒット時0.1x）を適用すると、コスト削減率はさらに大きくなる。

## 関連研究（Related Work）

- **PagedAttention / vLLM** (Kwon et al., SOSP 2023): KVキャッシュのメモリ管理を効率化するが、異なるリクエスト間でのKVキャッシュ再利用は限定的。本論文はこの制約を克服している
- **SGLang / RadixAttention** (Zheng et al., 2024): 基数木構造でプレフィックスキャッシュを管理するサービングシステム。本論文のPrompt Moduleはより細粒度なキャッシュ管理を提供する
- **CacheGen** (Liu et al., 2023): KVキャッシュの圧縮・ストリーミング転送。本論文のキャッシュ再利用とは補完的な関係にある

## まとめと今後の展望

著者らが提案するPrompt Cacheは、LLMのプロンプトプレフィックスをモジュール単位でKVキャッシュに保存・再利用することで、TTFTを最大8倍削減する推論時最適化手法である。モデル変更不要・精度劣化なしという実用上の利点があり、Anthropic・OpenAI等のプロダクション実装の理論的基盤となっている。

今後の課題として、マルチターン会話での中間状態キャッシュの最適化、キャッシュストレージのティアード管理（VRAM/RAM/SSD）の自動化が挙げられる。
