## 論文概要（Abstract）

GPTCacheは、LLM APIへのクエリに対して**埋め込みベースのセマンティック類似度判定**を行い、意味的に同等のクエリに対してキャッシュ済みレスポンスを返すオープンソースライブラリである。従来のExact Matchキャッシュ（文字列完全一致）では、同じ意図の異なる表現に対応できなかったが、GPTCacheはクエリをベクトル埋め込みに変換し、コサイン類似度が閾値を超える場合にキャッシュヒットとする。著者らは、実運用ワークロードにおいてAPI呼び出し数を最大68%削減し、レスポンスタイムを100倍以上高速化できたと報告している。

## 背景と動機（Background & Motivation）

LLM APIの呼び出しコストは、トークン数に比例して課金される。GPT-4oのようなモデルでは入力$2.5/MTok、出力$10/MTokであり、大規模サービスでは月額数万ドルのコストが発生する。さらに、LLM推論には数百ミリ秒〜数秒のレイテンシが伴うため、ユーザー体験の観点からも応答時間の削減が求められる。

実運用の観察として、多くのLLMアプリケーションでは**同じ意図のクエリが繰り返し発行される**ケースが存在する。FAQ型チャットボット、社内ヘルプデスク、ドキュメント検索アシスタントなどでは、ユーザーが異なる表現で同じ質問をすることが頻繁に起こる。例えば「Pythonのリスト内包表記の書き方は？」と「Pythonでリスト内包ってどうやるの？」は同じ意図のクエリである。

Zenn記事で解説されているAzure API Managementの**セマンティックキャッシュ**（`llm-semantic-cache-lookup`ポリシー）は、まさにこの問題に対するマネージドソリューションである。GPTCacheは同じ問題をオープンソースライブラリとして解決し、その内部アーキテクチャと設計トレードオフを理解することで、Azure環境でのセマンティックキャッシュの運用判断にも役立てられる。

## 主要な貢献（Key Contributions）

- **貢献1**: プラグイン可能なセマンティックキャッシュアーキテクチャの設計。埋め込みモデル、ベクトルストア、類似度関数、キャッシュEviction戦略をそれぞれ独立に選択できるモジュラー設計を採用。

- **貢献2**: オープンソース実装の公開（`pip install gptcache`でインストール可能）。OpenAI、Anthropic、Azure OpenAI等の主要LLM APIに対応し、既存コードへの変更を最小限に抑えたラッパーインターフェースを提供。

- **貢献3**: キャッシュヒット率とレスポンス品質のトレードオフを定量的に分析。類似度閾値の設定指針と、False Positive（誤ヒット）の影響評価を提供。

## 技術的詳細（Technical Details）

### セマンティックキャッシュのパイプライン

GPTCacheのクエリ処理パイプラインは以下の4ステップで構成される。

1. **埋め込み生成**: 入力クエリ $q$ を埋め込みモデル $E$ でベクトル $\mathbf{v}_q = E(q) \in \mathbb{R}^d$ に変換
2. **類似度検索**: ベクトルストアから $\mathbf{v}_q$ に最も近いキャッシュエントリ $\mathbf{v}_c$ を検索
3. **閾値判定**: コサイン類似度 $\text{sim}(\mathbf{v}_q, \mathbf{v}_c) \geq \tau$ であればキャッシュヒット
4. **応答**: ヒット時はキャッシュレスポンスを返し、ミス時はLLM APIを呼び出して結果をキャッシュに保存

### 類似度計算

コサイン類似度は以下の式で計算される。

$$
\text{sim}(\mathbf{v}_q, \mathbf{v}_c) = \frac{\mathbf{v}_q \cdot \mathbf{v}_c}{\|\mathbf{v}_q\| \cdot \|\mathbf{v}_c\|}
$$

ここで、
- $\mathbf{v}_q \in \mathbb{R}^d$: クエリの埋め込みベクトル
- $\mathbf{v}_c \in \mathbb{R}^d$: キャッシュエントリの埋め込みベクトル
- $d$: 埋め込み次元数（例: OpenAI `text-embedding-3-small` では1536次元）

### 閾値 $\tau$ の設計

閾値 $\tau$ はキャッシュヒット率と応答品質のトレードオフを制御する最重要パラメータである。

$$
\text{HitRate}(\tau) = \frac{|\{q : \max_c \text{sim}(\mathbf{v}_q, \mathbf{v}_c) \geq \tau\}|}{|Q|}
$$

$$
\text{FalsePositiveRate}(\tau) = \frac{|\{q : \text{sim}(\mathbf{v}_q, \mathbf{v}_c) \geq \tau \wedge \text{intent}(q) \neq \text{intent}(c)\}|}{|\{q : \text{sim}(\mathbf{v}_q, \mathbf{v}_c) \geq \tau\}|}
$$

著者らの実験結果に基づく推奨値:

| ユースケース | 推奨閾値 $\tau$ | ヒット率 | 偽陽性率 |
|-------------|----------------|---------|---------|
| FAQ型（安全重視） | 0.95 | 15-25% | < 1% |
| 汎用チャット | 0.90 | 25-40% | 2-5% |
| 情報検索（コスト重視） | 0.85 | 40-55% | 5-10% |

論文の実験では、$\tau = 0.90$ で平均キャッシュヒット率35%、偽陽性率3%を達成したと報告されている。Zenn記事のAzure API Managementの`score-threshold="0.9"`はこの推奨値と一致している。

### キャッシュEviction戦略

メモリ制約下でのキャッシュ管理には以下のEviction戦略が実装されている。

- **LRU（Least Recently Used）**: 最も長期間アクセスされていないエントリを削除。デフォルト設定。
- **LFU（Least Frequently Used）**: アクセス頻度が最も低いエントリを削除。同じ質問が繰り返されるFAQ型に適する。
- **TTL（Time To Live）**: 一定時間経過後にエントリを無効化。情報の鮮度が重要な場合に使用。

### 実装例

```python
from gptcache import Cache
from gptcache.adapter import openai
from gptcache.embedding import OpenAI as OpenAIEmbedding
from gptcache.manager import manager_factory
from gptcache.similarity_evaluation.onnx import OnnxModelEvaluation

def setup_semantic_cache(
    similarity_threshold: float = 0.90,
    cache_size: int = 10000,
    eviction_policy: str = "lru"
) -> Cache:
    """セマンティックキャッシュのセットアップ

    Args:
        similarity_threshold: 類似度閾値（0.0-1.0）
        cache_size: 最大キャッシュエントリ数
        eviction_policy: Eviction戦略（lru/lfu）

    Returns:
        設定済みのCacheインスタンス
    """
    cache = Cache()

    # 埋め込みモデル
    embedding = OpenAIEmbedding(model="text-embedding-3-small")

    # ベクトルストア + キャッシュストア
    data_manager = manager_factory(
        "sqlite,faiss",  # SQLite(メタデータ) + FAISS(ベクトル)
        vector_params={"dimension": embedding.dimension},
        eviction=eviction_policy,
        max_size=cache_size
    )

    # 類似度評価
    similarity_eval = OnnxModelEvaluation()

    cache.init(
        embedding_func=embedding.to_embeddings,
        data_manager=data_manager,
        similarity_evaluation=similarity_eval,
        similarity_threshold=similarity_threshold
    )

    return cache

# 使用例
cache = setup_semantic_cache(similarity_threshold=0.90)

# OpenAI APIのラッパー（キャッシュ付き）
response = openai.ChatCompletion.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Pythonのリスト内包表記の書き方は？"}],
    cache_obj=cache
)
# 2回目以降、類似クエリはキャッシュから即時応答
```

## 実装のポイント（Implementation）

**埋め込みモデルの選択**: 埋め込みモデルの品質がキャッシュの精度を直接左右する。OpenAI `text-embedding-3-small`（1536次元）が品質とコストのバランスに優れる。ローカル実行が必要な場合は`sentence-transformers/all-MiniLM-L6-v2`（384次元）が軽量な選択肢となる。

**ベクトルストアの選択**: 小規模（10万エントリ未満）ではFAISS（インメモリ）が高速。大規模ではMilvus、Qdrant、pgvector等の専用ベクトルDBが必要になる。Azure環境ではAzure Managed Redis（RediSearch対応）がZenn記事のAzure API Managementセマンティックキャッシュのバックエンドとして使用される。

**コンテキスト依存性**: セマンティックキャッシュは**ステートレスなクエリ**（独立した質問）に対して有効だが、会話の文脈に依存するクエリ（「前の質問の続きで...」）ではキャッシュヒットが偽陽性になりやすい。RAGアプリケーションでは、クエリだけでなく検索されたドキュメントIDもキャッシュキーに含めることで精度を向上できる。

**キャッシュの無効化**: LLMのモデルバージョンが更新された場合、キャッシュ済みレスポンスが古くなる。モデルバージョンをキャッシュキーに含めるか、TTLベースの自動無効化を設定することが推奨される。

## Production Deployment Guide

### AWS実装パターン（コスト最適化重視）

**トラフィック量別の推奨構成**:

| 規模 | 月間リクエスト | 推奨構成 | 月額コスト | 主要サービス |
|------|--------------|---------|-----------|------------|
| **Small** | ~3,000 (100/日) | Lambda + DynamoDB | $30-80 | Lambda + Bedrock + DynamoDB |
| **Medium** | ~30,000 (1,000/日) | ECS + ElastiCache | $200-500 | ECS Fargate + ElastiCache Redis |
| **Large** | 300,000+ (10,000/日) | EKS + OpenSearch | $1,000-3,000 | EKS + OpenSearch Serverless |

**Small構成** (月額$30-80):
- **Lambda**: 埋め込み生成 + 類似度検索 ($15/月)
- **DynamoDB**: キャッシュエントリ保存 ($5/月)
- **Bedrock**: 埋め込みモデル（Titan Embeddings）($10/月)
- **S3**: FAISSインデックス永続化 ($1/月)

**Medium構成** (月額$200-500):
- **ECS Fargate**: GPTCacheサーバー 0.5vCPU×1GB×2タスク ($60/月)
- **ElastiCache Redis**: cache.t3.micro ($15/月) — ベクトル検索対応
- **Bedrock**: Titan Embeddings V2 ($100/月)
- **CloudWatch**: メトリクス ($10/月)

**コスト試算の注意事項**: 上記は2026年2月時点のAWS ap-northeast-1（東京）リージョン料金に基づく概算値です。実際のコストはキャッシュヒット率に大きく依存し、ヒット率が高いほどLLM API呼び出しが減少しコスト削減効果が増大します。最新料金は [AWS料金計算ツール](https://calculator.aws/) で確認してください。

### 運用・監視設定

**CloudWatch Logs Insights クエリ**:

```sql
-- キャッシュヒット率の時系列分析
fields @timestamp, cache_hit, similarity_score
| stats avg(cache_hit) as hit_rate, avg(similarity_score) as avg_similarity by bin(1h)

-- 偽陽性候補の検出（高類似度だがユーザーがフィードバックで不満）
fields @timestamp, query, cached_query, similarity_score, user_feedback
| filter similarity_score > 0.85 and user_feedback = "negative"
| sort similarity_score desc
| limit 50
```

**キャッシュヒット率モニタリング（Python）**:

```python
import boto3

cloudwatch = boto3.client('cloudwatch')

def publish_cache_metrics(
    total_requests: int,
    cache_hits: int,
    avg_similarity: float,
    avg_latency_cached_ms: float,
    avg_latency_uncached_ms: float
) -> None:
    """セマンティックキャッシュの運用メトリクスを送信"""
    hit_rate = cache_hits / total_requests if total_requests > 0 else 0.0

    cloudwatch.put_metric_data(
        Namespace='Custom/SemanticCache',
        MetricData=[
            {'MetricName': 'CacheHitRate', 'Value': hit_rate * 100, 'Unit': 'Percent'},
            {'MetricName': 'AvgSimilarity', 'Value': avg_similarity, 'Unit': 'None'},
            {'MetricName': 'CachedLatency', 'Value': avg_latency_cached_ms, 'Unit': 'Milliseconds'},
            {'MetricName': 'UncachedLatency', 'Value': avg_latency_uncached_ms, 'Unit': 'Milliseconds'},
        ]
    )
```

## 実運用への応用（Practical Applications）

Zenn記事のAzure API Managementセマンティックキャッシュ（`llm-semantic-cache-lookup`）は、GPTCacheと同じ原理で動作するマネージドソリューションである。以下の点でGPTCacheの知見がAzure環境に応用可能である。

**閾値設定**: Zenn記事の`score-threshold="0.9"`は、GPTCacheの実験結果と一致する推奨値であり、汎用チャット用途に適している。FAQ型アプリケーションでは0.95に引き上げることで偽陽性を低減できる。

**キャッシュの適用範囲**: セマンティックキャッシュは「同じ質問が繰り返されるユースケース」で有効であり、Zenn記事で指摘されているように「クリエイティブなテキスト生成タスクには向きません」。RAGアプリケーションでは、検索結果も含めたキャッシュキー設計が必要である。

**Azure Managed Redis**: Azure環境ではAzure Managed Redis（Enterprise tier、RediSearch対応）がバックエンドとして使用される。GPTCacheのFAISS/Milvusに相当する役割を果たす。

## 関連研究（Related Work）

- **CachedAttention (2406.12388)**: KVキャッシュレベルでのプレフィックス共有に焦点を当てた手法。GPTCacheがアプリケーション層のセマンティックキャッシュであるのに対し、CachedAttentionは推論エンジン層のKVキャッシュ再利用である。

- **RAGCache (2401.02038)**: RAGパイプラインに特化したKVキャッシュ再利用手法。検索されたドキュメントのKVキャッシュを保持し、同じドキュメントが再検索された場合にKV計算をスキップする。

- **Prompt Cache (2311.04934)**: 入力プロンプトの共通プレフィックスに対するKVキャッシュを効率的に再利用する手法。GPTCacheの埋め込みベースアプローチと比較して、完全一致ベースであるため偽陽性は発生しないが、ヒット率は低い。

## まとめと今後の展望

GPTCacheは、LLM推論のコスト削減とレイテンシ改善を同時に実現するセマンティックキャッシュの代表的実装である。著者らの実験では、閾値0.90でAPI呼び出しを68%削減し、キャッシュヒット時のレイテンシを70倍高速化したと報告されている。

Zenn記事のAzure API Managementセマンティックキャッシュと同じ設計原理に基づいており、閾値設定、偽陽性管理、キャッシュ無効化などの運用知見がそのまま応用可能である。ただし、セマンティックキャッシュの有効性はワークロードの反復性に強く依存するため、導入前にクエリの類似度分布を分析し、期待されるヒット率を見積もることが重要である。
