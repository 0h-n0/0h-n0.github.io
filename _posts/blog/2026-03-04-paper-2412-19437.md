## 論文概要（Abstract）

DeepSeek-V3は、DeepSeek-AIが2024年12月に公開したMixture-of-Experts（MoE）言語モデルである。総パラメータ数671B、トークンあたりのアクティブパラメータ数37Bという構成で、オープンソースモデルとして当時最高水準の性能を達成したと報告されている。学習には14.8兆トークンを使用し、2.788M H800 GPU時間で完了した。学習過程全体を通じて回復不能なロススパイクやロールバックは発生しなかったと著者らは述べている。

## 背景と動機（Background & Motivation）

大規模言語モデル（LLM）のスケーリングにおいて、密な（Dense）アーキテクチャでは計算コストがパラメータ数に比例して増大するという根本的な問題がある。GPT-3（175B）やLlama 2（70B）のような密なモデルは、すべてのパラメータが各トークンの処理に関与するため、パラメータ数の増加が直接的にFLOPsの増大につながる。

DeepSeek-V3は、この問題に対してMoEアーキテクチャを採用し、671Bの総パラメータのうち37B（約5.5%）のみを各トークンでアクティブにすることで、密な37Bモデルと同等の推論コストで大幅に高い性能を実現することを目指した。さらに、従来のMoE学習における2つの大きな課題——負荷分散のための補助損失がモデル性能を低下させる問題と、学習の不安定性——に対して新しい解決策を提案している。

## 主要な貢献（Key Contributions）

著者らが報告する主要な貢献は以下の通りである：

- **Auxiliary-Loss-Free Load Balancing**: 補助損失関数なしでエキスパート負荷分散を実現する手法。ルーティングスコアにバイアス項を追加し動的に調整する
- **Multi-head Latent Attention (MLA)**: KVキャッシュを圧縮し推論効率を向上させる注意機構。DeepSeek-V2で導入された手法の改良版
- **FP8混合精度学習**: 学習の大部分をFP8精度で実行し、メモリ使用量と計算コストを削減。著者らによると、BF16学習と比較して精度低下なしで学習を加速
- **Multi-Token Prediction (MTP)**: 1トークンの予測に加え、後続の複数トークンも同時に予測する学習目標。推論時のSpeculative Decodingにも活用可能

## 技術的詳細（Technical Details）

### アーキテクチャ全体構成

DeepSeek-V3のTransformerは61層から構成される。最初の3層は通常の密なFFN層を使用し、残りの58層がMoE層となる。各MoE層は**1個の共有エキスパート**と**256個のルーテッドエキスパート**を持ち、各トークンは共有エキスパートに加えてTop-8のルーテッドエキスパートを選択する。

$$
y = E_{\text{shared}}(x) + \sum_{i \in \text{Top8}} g_i \cdot E_i(x)
$$

ここで、
- $x$: 入力トークンの隠れ表現（$d_{\text{model}} = 7168$）
- $E_{\text{shared}}$: 共有エキスパート（全トークンで常にアクティブ）
- $E_i$: $i$番目のルーテッドエキスパート
- $g_i$: トークン$x$に対するエキスパート$i$のゲーティングスコア

### Auxiliary-Loss-Free Load Balancing

従来のMoE学習では、ルーティング崩壊（特定のエキスパートにトークンが集中する現象）を防ぐために補助損失関数を使用していた。Switch Transformer（Fedus et al., 2021）では以下の補助損失が使用される：

$$
L_{\text{aux}} = \alpha \cdot N \sum_{i=1}^{N} f_i \cdot P_i
$$

ここで$f_i$はエキスパート$i$に割り当てられたトークンの割合、$P_i$はルーティング確率の平均、$\alpha$は補助損失の重み係数である。

この方式の問題は、$\alpha$が大きすぎるとモデルの表現力が損なわれ、小さすぎると負荷分散が不十分になるというトレードオフにある。

DeepSeek-V3はこの問題を、**ルーティングスコアにバイアス項を追加し、学習中にバイアスを動的に調整する**方式で解決している。具体的には：

1. 各エキスパートにバイアス項 $b_i$ を持つ（学習対象外の変数）
2. **エキスパート選択時**には $s_i + b_i$ を使用（バイアス込み）
3. **ゲーティング値の計算時**には元のスコア $s_i$ のみ使用（バイアスなし）
4. 各学習ステップで、過負荷エキスパートの $b_i$ を減少、低負荷エキスパートの $b_i$ を増加

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class DeepSeekV3Router(nn.Module):
    """DeepSeek-V3のAuxiliary-Loss-Free Router

    バイアス項によるエキスパート選択と、元のアフィニティスコアによる
    ゲーティング値計算を分離する。
    """

    def __init__(self, d_model: int, num_experts: int, top_k: int = 8):
        super().__init__()
        self.top_k = top_k
        self.num_experts = num_experts
        self.gate = nn.Linear(d_model, num_experts, bias=False)
        # 負荷分散バイアス（学習対象外）
        self.expert_bias = nn.Parameter(
            torch.zeros(num_experts), requires_grad=False
        )
        self.gamma = 0.001  # バイアス更新係数

    def forward(self, x: torch.Tensor) -> tuple[torch.Tensor, torch.Tensor]:
        """ルーティングスコアを計算し、TopKエキスパートを選択する

        Args:
            x: 入力テンソル (batch, seq, d_model)

        Returns:
            top_k_indices: 選択されたエキスパートのインデックス
            gate_values: ゲーティングスコア（正規化済み）
        """
        # アフィニティスコア計算
        affinity = self.gate(x)  # (batch, seq, num_experts)

        # エキスパート選択にはバイアスを加算
        biased_scores = affinity + self.expert_bias
        top_k_indices = biased_scores.topk(self.top_k, dim=-1).indices

        # ゲーティング値は元のアフィニティから計算
        gate_values = torch.zeros_like(affinity)
        gate_values.scatter_(-1, top_k_indices,
                            affinity.gather(-1, top_k_indices))
        gate_values = F.softmax(
            gate_values.masked_fill(gate_values == 0, float('-inf')),
            dim=-1
        )

        return top_k_indices, gate_values.gather(-1, top_k_indices)

    def update_bias(self, expert_load: torch.Tensor):
        """学習ステップごとにバイアスを更新

        Args:
            expert_load: 各エキスパートの現在の負荷率
        """
        target = 1.0 / self.num_experts
        self.expert_bias[expert_load > target] -= self.gamma
        self.expert_bias[expert_load < target] += self.gamma
```

この設計の重要なポイントは、**バイアスがエキスパート選択のみに影響し、ゲーティング値には影響しない**ことである。これにより、エキスパートの専門性を保ちながら負荷分散を実現できる。著者らの報告によれば、この方式により従来の補助損失ベースの方式と比較して、同一計算量で有意に優れた性能を達成している。

### Multi-head Latent Attention (MLA)

MLA はDeepSeek-V2で導入された注意機構で、KV（Key-Value）キャッシュの圧縮を実現する。標準的なMulti-Head Attention (MHA) では、各ヘッドが独立したK, Vベクトルを保持するため、KVキャッシュのサイズは $2 \times n_h \times d_h \times L$ となる（$n_h$: ヘッド数、$d_h$: ヘッド次元、$L$: 系列長）。

MLAでは、K, Vの情報を低次元の潜在表現（latent representation）に圧縮する：

$$
c_t^{KV} = W^{DKV} h_t \in \mathbb{R}^{d_c}
$$

$$
K_t = W^{UK} c_t^{KV}, \quad V_t = W^{UV} c_t^{KV}
$$

ここで、
- $h_t$: 位置$t$の隠れ状態（$d_{\text{model}} = 7168$）
- $d_c$: 圧縮後の次元（512、元の次元の約7%）
- $W^{DKV}$: ダウンプロジェクション行列
- $W^{UK}, W^{UV}$: アッププロジェクション行列

KVキャッシュには圧縮された $c_t^{KV}$ のみを保持すればよいため、メモリ使用量がMHAの約1/14に削減される。DeepSeek-V3では $n_h = 128$、$d_h = 128$ に対して $d_c = 512$ であり、著者らによるとGQA（Grouped Query Attention）と比較しても推論効率と性能の両面で優位であると報告されている。

### FP8混合精度学習

DeepSeek-V3では、学習パイプラインの大部分をFP8精度で実行する混合精度学習を採用している。具体的には：

- **FP8で実行**: 行列乗算（GEMM）の主要部分
- **BF16/FP32で実行**: 正規化層、ゲーティングスコアの計算、損失関数の計算

FP8の動的範囲は限られている（E4M3形式で $\pm 448$）ため、テンソルごとの細粒度スケーリングが必要となる。DeepSeek-V3では、128×128のタイル単位でスケーリング係数を計算する方式を採用し、精度の損失を抑制している。

著者らの報告では、FP8学習はBF16学習と比較して最終的なモデル性能に有意な差がなく、メモリ使用量と計算コストの両面で削減を達成したとされている。

### Multi-Token Prediction (MTP)

従来のLLM学習では、各位置で次の1トークンのみを予測する。DeepSeek-V3では、追加のMTPヘッドを使用して後続の複数トークンも同時に予測する学習目標を導入している。

$$
L_{\text{total}} = L_{\text{next-token}} + \lambda \sum_{k=1}^{K} L_{\text{MTP}}^{(k)}
$$

ここで$K$はMTPの予測深度（DeepSeek-V3では$K=1$、つまり次の次の1トークンを追加予測）、$\lambda$は重み係数である。

MTPの効果として著者らが報告しているのは：
1. 学習時のデータ効率の向上（同一トークン数でより多くの学習信号を得る）
2. 推論時のSpeculative Decodingへの活用（MTPヘッドがドラフトトークンを生成）

## 実装のポイント（Implementation）

DeepSeek-V3の推論デプロイにおける主要な実装上の考慮点：

**Expert Parallelism**: 256個のルーテッドエキスパートをGPU間で分散配置するため、all-to-all通信が必要となる。各トークンのルーティング決定に基づき、適切なGPUへトークンを転送する。DeepSeek-V3ではこの通信コストを最小化するため、エキスパートの配置をルーティングパターンに最適化している。

**KVキャッシュの最適化**: MLAにより圧縮されたKVキャッシュ（$d_c = 512$）を活用することで、長いコンテキスト（128Kトークン）でもメモリ効率を維持できる。ただし、デコード時にはアッププロジェクション($W^{UK}, W^{UV}$)の計算が追加で必要となる。

**FP8推論**: 学習時にFP8を使用しているため、推論時もFP8精度で実行可能。NVIDIA H100/H200のFP8 Tensor Coreを活用することで、BF16推論と比較して約2倍のスループットが期待できる。

**デプロイ時の注意点**: 671Bの総パラメータはFP8でも約670GBのメモリを必要とする。8台のH100（80GB）でぎりぎり収まるが、KVキャッシュやアクティベーション用のメモリも考慮すると、16台以上のGPUが推奨される。

## Production Deployment Guide

### AWS実装パターン（コスト最適化重視）

DeepSeek-V3のような大規模MoEモデルのデプロイは、一般的なLLMと比較してメモリ要件が特殊である。総パラメータ671Bを保持しつつ、37Bのアクティブパラメータで推論を行う。

**トラフィック量別の推奨構成**:

| 規模 | 月間リクエスト | 推奨構成 | 月額コスト | 主要サービス |
|------|--------------|---------|-----------|------------|
| **Small** | ~3,000 (100/日) | Serverless | $200-500 | Lambda + Bedrock (Claude/Llama) |
| **Medium** | ~30,000 (1,000/日) | Self-hosted | $3,000-8,000 | ECS + p5.48xlarge × 2 |
| **Large** | 300,000+ (10,000/日) | Cluster | $15,000-40,000 | EKS + p5.48xlarge × 4-8 |

**注意**: DeepSeek-V3（671B）の自前ホスティングにはH100 GPUが最低8台必要であり、Small構成ではBedrock経由で他のMoEモデル（例: Mixtral）を利用する方がコスト効率が高い。

**コスト試算の注意事項**:
- 上記は2026年3月時点のAWS ap-northeast-1（東京）リージョン料金に基づく概算値
- 実際のコストはトラフィックパターン、リージョン、バースト使用量により変動
- 最新料金は [AWS料金計算ツール](https://calculator.aws/) で確認推奨

**コスト削減テクニック**:
- Spot Instances使用で最大90%削減（p5インスタンス、ただしGPUインスタンスのSpot可用性に注意）
- Reserved Instances購入で最大72%削減（1年コミット）
- FP8推論によりGPU台数を半減可能（FP8対応ハードウェア使用時）
- vLLMのTensor Parallelism + Expert Parallelismでスループットを最適化

## 実運用への応用（Practical Applications）

DeepSeek-V3の設計から得られる実務上の示唆：

**MoEアーキテクチャの採用判断**: 推論レイテンシの改善が必要で、かつメモリ（VRAM）に余裕がある場合、MoEは有力な選択肢となる。ただし、全パラメータのメモリ保持が必要なため、モデルのロード時間とVRAM要件は密なモデルより大きくなる。

**負荷分散手法の選択**: Auxiliary-Loss-Free方式はモデル性能への影響が小さいとされるが、バイアスの更新頻度やスケールのチューニングが必要。小規模なMoEモデルでは従来の補助損失方式で十分な場合もある。

**FP8学習の検討**: H100以降のGPUを使用する場合、FP8学習は学習コストの削減に有効。ただし、細粒度スケーリングの実装が必要であり、フレームワークのサポート状況（DeepSpeed、Megatron-LM等）を確認する必要がある。
