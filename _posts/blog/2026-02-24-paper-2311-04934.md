本記事は [Prompt Cache: Modular Attention Reuse for Low-Latency Inference（arxiv:2311.04934）](https://arxiv.org/abs/2311.04934) の解説記事です。

## 論文概要（Abstract）

Prompt Cacheは、LLM推論においてプロンプト内の繰り返し出現するテキストセグメント（システムメッセージ、プロンプトテンプレート、ドキュメントコンテキスト等）のAttention Stateを事前計算・保存し、後続リクエストで再利用するシステムである。著者らは、プロンプトを再利用可能な「モジュール」として定義するPrompt Markup Language（PML）スキーマを提案し、GPU推論でTTFT最大8倍、CPU推論で最大60倍の高速化を報告している。

## 背景と動機（Background & Motivation）

LLMの推論において、同一のシステムプロンプトやテンプレートが多数のリクエストで繰り返し使用されるにもかかわらず、各リクエストでAttention計算（KVテンソルの生成）をゼロから実行するのは計算資源の大きな浪費である。特にドキュメントQAやRAGのように長大なコンテキストを含むプロンプトでは、TTFTがユーザー体験のボトルネックとなる。

従来のプレフィックスキャッシュ（vLLM等）は「先頭からの完全一致」のみをキャッシュ対象とするため、プロンプト内の任意のセグメントを柔軟に再利用できない制約があった。著者らはこの課題に対して、プロンプトを独立したモジュールとして定義・管理し、位置情報を保ったまま任意のセグメントのAttention Stateを再利用する手法を提案している。

## 主要な貢献（Key Contributions）

- **貢献1**: Prompt Markup Language（PML）の設計。プロンプト内の再利用可能なテキストセグメントを「モジュール」として定義するスキーマを提案。モジュール間の位置整合性を保証する仕組みを導入
- **貢献2**: Attention State（KVテンソル）のモジュール単位での事前計算・キャッシュ・再利用メカニズム。プレフィックス完全一致の制約を解消し、プロンプト内の任意位置のセグメントを再利用可能に
- **貢献3**: モデルパラメータの変更なしで、GPU推論でTTFT最大8倍、CPU推論で最大60倍の高速化を実現

## 技術的詳細（Technical Details）

### Prompt Markup Language（PML）

PMLは、プロンプト内の再利用可能なセグメントを宣言的に定義するスキーマである。各モジュールにはテキスト内容と**固定の位置（position）**が割り当てられる。この位置情報が、Attention Stateの再利用時に位置エンコーディング（Positional Encoding）との整合性を保証する鍵である。

```xml
<prompt>
  <module name="system_instructions" position="0">
    あなたは社内ドキュメント検索AIです。以下のルールに従って回答してください...
  </module>
  <module name="tool_definitions" position="1024">
    {ツール定義のテキスト}
  </module>
  <module name="document_context" position="2048">
    {RAGで取得したドキュメント}
  </module>
  <variable name="user_query" position="4096">
    {ユーザーの質問}
  </variable>
</prompt>
```

### Attention Stateの事前計算と再利用

TransformerのSelf-Attentionにおいて、入力トークン列$\mathbf{x} = (x_1, x_2, \ldots, x_n)$に対するKVテンソルは以下で計算される。

$$
\mathbf{K} = \mathbf{x} \mathbf{W}_K, \quad \mathbf{V} = \mathbf{x} \mathbf{W}_V
$$

ここで、
- $\mathbf{K} \in \mathbb{R}^{n \times d_k}$: Key行列
- $\mathbf{V} \in \mathbb{R}^{n \times d_v}$: Value行列
- $\mathbf{W}_K, \mathbf{W}_V$: 学習済みの射影行列

Prompt Cacheでは、モジュール$m$のトークン列$\mathbf{x}_m$に対するKVテンソル$(\mathbf{K}_m, \mathbf{V}_m)$を事前計算してキャッシュに保存する。

**位置エンコーディングの整合性**: RoPEを使用するモデルでは、Keyテンソルに位置依存の回転が適用される。

$$
\mathbf{K}_i = \mathbf{R}(i) \cdot \mathbf{x}_i \mathbf{W}_K
$$

ここで$\mathbf{R}(i)$は位置$i$に対する回転行列である。モジュールの位置を固定するため、事前計算されたKVテンソルの位置情報がリクエスト間で一致し、再利用時に不整合が生じない。

### アルゴリズム

```python
from dataclasses import dataclass
from typing import Optional

import torch


@dataclass
class PromptModule:
    """プロンプトモジュールの定義"""
    name: str
    text: str
    position: int
    kv_cache: Optional[tuple[torch.Tensor, torch.Tensor]] = None


class PromptCacheEngine:
    """Prompt Cacheの推論エンジン"""

    def __init__(self, model: torch.nn.Module):
        self.model = model
        self.cache: dict[str, tuple[torch.Tensor, torch.Tensor]] = {}

    def precompute_module(self, module: PromptModule) -> None:
        """モジュールのKVテンソルを事前計算してキャッシュに保存する。

        Args:
            module: 事前計算対象のプロンプトモジュール
        """
        tokens = self.model.tokenizer.encode(module.text)
        position_ids = torch.arange(
            module.position, module.position + len(tokens)
        )
        with torch.no_grad():
            kv = self.model.compute_kv(tokens, position_ids)
        self.cache[module.name] = kv

    def inference(
        self,
        modules: list[PromptModule],
        query_tokens: list[int],
        query_position: int,
    ) -> torch.Tensor:
        """キャッシュを活用した推論を実行する。

        Args:
            modules: 使用するモジュールのリスト
            query_tokens: クエリ（動的部分）のトークン
            query_position: クエリの開始位置

        Returns:
            生成されたトークンのテンソル
        """
        cached_kvs = []
        uncached_tokens = []
        uncached_positions = []

        for mod in modules:
            if mod.name in self.cache:
                cached_kvs.append(self.cache[mod.name])
            else:
                tokens = self.model.tokenizer.encode(mod.text)
                positions = list(range(
                    mod.position, mod.position + len(tokens)
                ))
                uncached_tokens.extend(tokens)
                uncached_positions.extend(positions)

        all_kv = self._merge_kv(
            cached_kvs, uncached_tokens, uncached_positions
        )
        return self.model.generate(
            query_tokens, query_position, past_kv=all_kv
        )
```

## 実装のポイント（Implementation）

### 位置の固定化とスロット設計

1. **スロット設計**: 各モジュールの位置を十分な間隔で配置し、モジュール長の変動に対応できるようにする。例えば、システムプロンプト用に位置0-1023、ツール定義用に1024-2047を予約
2. **パディング**: モジュールのトークン数がスロットサイズより短い場合、残り部分をパディングトークンで埋める。Attention Maskの整合性を保つ
3. **キャッシュの無効化**: モジュール内容が更新された場合、該当モジュールのKVキャッシュのみを再計算する（他のモジュールは影響を受けない）

### 既存APIとの関連

- **Anthropicの`cache_control`ブレークポイント**: PMLのモジュール境界に相当。最大4つのキャッシュポイント配置はモジュール概念の簡略化と解釈できる
- **OpenAIの自動プレフィックスキャッシュ**: PMLの位置固定を暗黙的に実現
- **Geminiの明示的キャッシュオブジェクト**: PMLのモジュールに相当するキャッシュオブジェクトをAPI経由で管理

## 実験結果（Results）

著者らはLLaMAモデルを用いて評価を行っている（論文Table 2, Table 3より）。

| タスク | プロンプト長 | GPU TTFT改善 | CPU TTFT改善 | 出力品質 |
|--------|------------|-------------|-------------|---------|
| ドキュメントQA | ~4,000トークン | 5.4x | 43x | 劣化なし |
| 推薦システム | ~8,000トークン | 8x | 60x | 劣化なし |
| コード生成 | ~2,000トークン | 3.2x | 21x | 劣化なし |

著者らは、プロンプト長が長いほどTTFT改善効果が大きくなると報告している。CPU推論での改善幅がGPUより大きい理由として、GPU推論はメモリ帯域に律速されるため計算スキップのメリットが相対的に小さくなると分析している。

## まとめと今後の展望

Prompt Cacheは、PMLスキーマと位置整合性保証により、プレフィックス完全一致の制約を超えた柔軟なAttention State再利用を実現した。モデル変更不要でGPU 8倍・CPU 60倍のTTFT改善は、実運用への導入障壁が低い。今後はRoPE以外の位置エンコーディング対応、動的モジュール長の効率化、分散環境でのキャッシュ管理が課題として挙げられている。
